\section{Motivation}
Many aspects of our day-to-day lives revolves around finding
optimality in problems that we can have no comprehensible insight
about. This abstraction could be due to several factors. For
instance, optimizing a design value in many large-scale and complex
systems proves computationally or materially expensive. It can be
observed in industrial processes of !TODO!. These
types of problems are said to be \acrfull{bbo} problems.

We do not possess many tools to interact with them save from a
cost/objective function $\mathbf{f}: \mathcal{X} \mapsto \mathcal{S}$
(the `oracle') that provides an abstract representation of the
problem, where $\mathcal{X}$ is the search space, $\mathcal{S}$ the
evaluation space !TODO: find concrete refs!. Typically, $\mathcal{S}
\subseteq R$. The task is to minimize or maximize the solutions
produced by this objective function, i.e., find a solution that Much
study is done in optimizing this cost function without the liberty of
classic mathematical tools such as continuity, smoothness and
convexity (these methods warrant the oracle to return additional
  information, such as gradients, apart from the evaluation value of an
input \cite{liu_primer_2020}).

While this approach secures the bounds on the first sampling of the
optimum, it presents no clues on the solution quality produced by the
algorithm. Consider the case when the algorithm samples the optimum
at time $t$, then in the next step $t+1$, moves completely away from
the vicinity. This algorithm might then produce worse answers, asymptotically.

\subsection{Conventional Solutions}
% A brief history of how this problem came to be
Classical analysis of solving \acrshort{bbo} problems are concerned
with the so-called \acrfull{bbc}, that is, deriving the runtime
bounds for an algorithm to sample a (global) optimum, depending only
on the problem size $\mathbf{n}$. The reasoning behind this is
palpable; the \textit{computational} complexity of evaluating the
objective function for such \acrshort{bbo} systems are usually expensive.

A trivial example is Hyperparameter Tuning in \acrfull{ml}
models~\cite{bischl2017mlrmbo}. Searching for optimal hyperparameters
is a \acrfull{bbo} problem that
optimizes a single or multiple objectives on the provided (training
and validation) data.
(e.g., accuracy, F1-score, $R^2$ score.).

\section{Report Structure}
This report is structured as follows:
\paragraph{Background [Chapter~\ref{cha:bg}]} compiles the necessary
knowledge sufficient to grasp the intent and work presented in the later texts.
\paragraph{Runtime Profiles and Anytime Performance
[Chapter~\ref{cha:profiles}]} delves into the core theory developed
this project, coupling a measure of regret with the metric of
`Runtime Profiles'.
\paragraph{Supplementary Software Design [Chapter~\ref{cha:design}]}
documents the novel, supplementary experimental frameworks developed
for this project. This software facilitated the experimentation and
exploratory analysis of various standard \acrshort{bbo} problems
\paragraph{Methodology [Chapter~\ref{cha:exper}]} inscribes the
exploratory and verification (based on concepts in
Chapter~\ref{cha:profiles}) experiments coordinated in this project;
emphasis is provided on specific rationale and decisions taken for
motivating the exploration.
\paragraph{Evaluation and Discussion of Results
[Chapter~\ref{cha:results}]} confers the results gathered from the
experiments conducted. Pointwise regret metrics extracted from the
experiments (against known optimums) are analyzed qualitatively and
quantitatively (especially for the verification experiments mentioned earlier).
\paragraph{Conclusion and Future Work [Chapter~\ref{cha:bg}]}
collates the substance, contributions, and achievements of the
project. Reflections and potential future directions beyond the
seminal work in this project are mapped out.
